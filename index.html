Activation layer · sequential after capability audit; concurrent tracks once prototype is live
Outcome engineering
Map data to decisions, not reports
Domain fluency meets product thinking. Owns the question: what must this system actually do for this person on a Monday morning?

  • Translate business outcomes into system behaviors
  • Define what "correct" looks like for each decision type
  • Set the bar for signal vs. noise
Artifact re-imagination
From audit-ready to agent-ready
Every organizational artifact must be re-examined for agent operationalizability:

  • Data definitions → meaning in context
  • Taxonomies → nodes and edges
  • Policies → decision hierarchies with lineage
  • Reports → living signals that route action
  • Contracts → active obligation graphs
  • KPIs → early warning signal systems
Workflow activation
Right action, right person, right time
Process maps written for human compliance become agent orchestration specs:

  • Triggers: what fires the workflow
  • Permissions: who can approve what, at which threshold
  • Escalation paths: encoded, not assumed
  • Decision lineage: every branch auditable
Prototype sprint
Working artifact, not a document
Sequential after the capability audit — cannot begin without knowing what the data reality actually is. Produces a functional mock against real data in days, not months. Becomes the RFP: you ask vendors "can you match or extend this?" not "can you interpret this spec?"
Trust architecture
Make the black box legible
Agentic systems only get adopted when operators can see why a recommendation was made. Decision lineage must be human-readable, not just machine-auditable. Concurrent with workflow activation — neither can ship without the other.
Gate 2 · Expand / contract
Evidence required: adoption rate + decision accuracy across pilot cohort
Technical infrastructure · capability audit first; all else concurrent once ontology is grounded
Capability audit
Must happen first. Everything gates on this.
Connect to existing systems. Extract signal about where operational friction lives. Produce a capability gap map grounded in actual data flows — not stated preferences. This becomes the anchor for every vendor conversation and the foundation for the ontology.
Ontology + knowledge graph
Entities, edges, meaning
Taxonomies are trees. Agents need graphs. Building the ontology forces the organization to answer questions it has never asked explicitly: what does "material" mean? What is the exact threshold for CFO approval? These governance gaps surface here — and must be resolved, not deferred.
Integration hub
Connect what exists, don't replace it
Yardi, MRI, Excel, SharePoint, email — the sources of operational truth. The integration hub ingests, normalizes, and routes. Concurrent with the ontology build. The ontology determines what fields mean; the hub determines where they come from.
Agent-ready data layer
Sequential after ontology is grounded
Audit-ready ≠ agent-ready. Data lineage ≠ decision lineage. The data layer must be structured for inference and grounded for decisions: context-scoped, temporally aware, and traceable to the human judgment it encodes. Cannot be built until the ontology establishes what entities and relationships exist.
Front end + UX
Interface as operating room
Not a dashboard. Not a new tool. A surface designed around decisions, not data. The asset manager sees what they need to act on — not everything the system knows. Concurrent with the agent-ready data layer. Neither ships without the other.
