00 · The argument
The old procurement motion is broken. Here’s what replaces it.

Historically, companies would hire a consulting firm to perform an assessment — detailed discovery interviews, documented functional requirements, translated to technical features, drafted into a request for information or proposal, run the vendor process, discovered the gaps that still needed to be built in-house, then run another process to find a partner to fill those gaps. Months and months pass. Once a solution is scoped and selected, the initial requirements have shifted. The process was designed for a world where building was slow and language was the only instrument of alignment.

That world is gone. In the age of AI, vibe coding, and forward-deployed engineers, the fundamental constraint has changed. Infrastructure that used to take six months to prototype now takes days. The bottleneck is no longer build speed — it’s clarity. And clarity cannot be manufactured through a requirements document. It has to be discovered through working software.

The prototype is not a deliverable. It’s a discovery instrument. The moment stakeholders react to working software instead of answering abstract questions, you learn more in one hour than six weeks of interviews would surface.

New motion 01
Prototype-first discovery

Instead of documenting requirements, build a working skeleton in days using vibe coding tools and forward-deployed engineers, then use it as the discovery instrument.

  • A small team (1–2 engineers + 1 domain expert) builds a functional mock against real data from one asset, wiring a few core workflows, surfacing the most contested questions in the interface itself
  • Stakeholders react to working software instead of answering abstract questions
  • The prototype becomes the RFP — you ask vendors “can you match or extend this?” not “can you interpret this 40-page spec?”
What this collapses: the entire discovery → requirements → RFP translation chain. That chain exists because language is lossy. Code is not.
New motion 02
Capability audit inversion

The traditional process starts with what users say they need. The new motion starts with what the data reveals is actually happening.

  • Before any vendor conversation, run a structured data audit using an AI layer — connect to Yardi, MRI, Excel, SharePoint, email — extract signal about where operational friction lives, where decisions stall, where reconciliation fails
  • The output is not a requirements document. It’s a capability gap map grounded in actual data flows, not stated preferences
  • This becomes the anchor for every vendor conversation: “here is our data reality, show us what you can do with it”
What this collapses: the “we need a consultant to tell us what we need” loop.
The three lanes and why the middle is the only unsolved one
Top lane · has a craft tradition
Asset managers, portfolio managers, investor relations
Their vocabulary is outcomes: protect yield, reduce leakage, win the next fund raise. They don’t think in data. They think in consequences. This discipline is centuries old and well-understood.
Middle lane · no craft tradition exists here
The activation layer — where the new work lives
What exists in the middle today is a combination of things that approximate it without naming it: business analysts who write requirements docs, project managers who run implementations, power users who build Excel frankenmodels that become load-bearing infrastructure, consultants who deliver decks that never get operationalized. None of these roles are the thing.
This is the actual gap. And it’s almost entirely unsolved.
Bottom lane · has a craft tradition
Database engineers, solutions architects, integration specialists
People have been doing this work for decades. The tools are mature: Yardi, MRI, APIs, cloud warehouses. The discipline is well-understood even when execution is hard.

What the middle actually requires is a different kind of practitioner — someone who can hold all three questions simultaneously: what decision does this data need to drive, how do we get the system to surface that signal reliably, and how do we make that legible to a person who has 90 seconds on a Monday morning.

That’s outcome engineering. It’s not analytics, it’s not PM, it’s not data engineering. It’s closer to product design for operational workflows — but specifically in a domain where the stakes of a wrong decision are measured in hundreds of millions of dollars.

AI and vibe coding lower the floor of the bottom lane dramatically. What used to require a six-month integration project can now be prototyped in days. That compression doesn’t eliminate the middle — it makes it the only remaining constraint. When infrastructure becomes cheap and fast to assemble, the bottleneck becomes: does anyone actually know what it needs to do and for whom?

That’s the design problem. And it’s still almost entirely unsolved.

02 · Artifact re-imagination
What organizations have vs. what agents need
Every artifact in a real estate organization was built for a reviewer, an auditor, or a regulator. None were built for a machine that must act on them. The shift from audit-era to agent-era requires re-meaning every artifact — not just reformatting it.
Data readiness — audit-ready ≠ agent-ready
The most fundamental shift in the entire program
TodayMust become
Audit-ready data — traceable, complete, defensible to a regulator→Agent-ready data — structured for inference, grounded, decision-scoped
Data lineage — where did this number come from?→Decision lineage — why did the system recommend this action?
Definition of data — what a field is called and how it’s typed→Meaning of data — what a value implies in context, for whom
Taxonomies & data dictionaries — labels → nodes and edges
Hierarchy is a tree. Agents need a graph.

A taxonomy feels structured — it has hierarchy, categories, controlled vocabulary. But hierarchy is a tree, and trees are one-dimensional relationships. What an agent needs is a graph: this asset class relates to this risk profile relates to this approval threshold relates to this reporting obligation.

The moment you add a second dimension of relationship, a taxonomy becomes an ontology, and the work required multiplies by an order of magnitude. Organizations that feed a flat taxonomy to an agent will get agents that confidently traverse the wrong paths.

  • Today: Classification trees — static, categorical, maintained in Excel or a data dictionary
  • Must become: Traversable graph with weighted relationships — machine-readable, queryable, version-controlled
Ontologies — entity model → people-in-the-loop context diagram
Who touches each entity, when, with what authority

Most enterprise ontologies capture what entities exist and how they relate technically. They do not capture who touches each entity, in what context, under what authority, and for what purpose.

An agent-ready ontology must include the human layer: the asset manager who reviews this entity every Monday, the approval threshold above which the CFO is required, the context that makes a given data value meaningful vs. noise.

  • Today: Assets, funds, entities — typed and related in a schema
  • Must become: People-in-the-loop context diagrams — who touches each entity, when, why, and with what authority
Policies & procedures — documented rules → decision hierarchies with lineage
An agent cannot exercise discretion. It needs explicit branch conditions.

A policy document is written in natural language, optimized for human interpretation, deliberately ambiguous at the edges to give practitioners discretion. An agent cannot exercise discretion — it needs a decision tree with explicit branch conditions, thresholds, exception paths, and escalation triggers.

Rewriting a policy as a decision hierarchy forces the organization to answer questions it has never had to make explicit: what does “material” mean exactly? What is the threshold at which CFO approval is required? The artifact transformation reveals governance gaps hidden in discretion.

  • Today: Written for human compliance — narrative, interpretable, auditable after the fact
  • Must become: Machine-executable decision hierarchies — every branch has a condition, every escalation has a trigger, every outcome is auditable
Contracts & agreements — document of record → active obligation graph
Clauses as live rules, not PDFs that surface when something lapses

A lease is not just a document — it is a set of time-triggered obligations, option windows, co-tenancy clauses, and renewal rights that currently live in a PDF and surface only when someone remembers to look, or when they don’t and something lapses.

An active obligation graph means every clause that has a downstream consequence is a live node that can fire. This is not a data problem. It’s an epistemology problem — you have to decide what the contract means operationally before you can encode it.

  • Today: Stored, searched when a dispute arises, reviewed at renewal
  • Must become: Clauses as live rules triggering downstream actions — expiry alerts, approval workflows, reporting obligations
Reporting & KPIs — static scorecard → early warning signal system
From backward-looking measurement to forward-looking prompts
  • Reports today: Snapshot in time, consumed and filed. Produced for a reviewer.
  • Reports must become: Living signals that trigger action and route to the right decision maker at the right moment
  • KPIs today: Measurement of what happened last period — lagging, descriptive, backward-looking
  • KPIs must become: Leading indicators that prompt before failure — forward-looking, actionable, personalized by role
  • Org charts today: Reporting lines and accountability on paper
  • Org charts must become: Decision authority maps — who can approve what, at which threshold, in which context
Workflows & process maps — steps for humans → agent orchestration specs
Triggers, permissions, escalation paths, and lineage — all encoded explicitly

Process maps documented for human execution cannot be handed to an agent. They are missing the four things an agent requires: explicit triggers that fire the workflow, permission boundaries at each step, escalation paths when conditions are not met, and decision lineage at every branch point.

  • Today: Steps documented for human execution — narrative, assumed context, discretion at every edge case
  • Must become: Agent orchestration specs — triggers, permissions, escalation paths, and lineage encoded explicitly
03 · Resourcing model
New roles, not the old org chart with new titles
The instinct is to solve a new problem with the old org chart. That’s where these programs die. The four archetypes below are not job titles you can post on LinkedIn and fill — they are skill clusters that currently exist partially, in scattered people who built them through unusual career paths.
What no longer works
Subject matter expert
Knows the domain, can’t build
Business analyst
Translates, can’t decide
Solutions architect
Designs, can’t operationalize
Project manager
Tracks, doesn’t shape
Vendor impl. consultant
Configures their product, leaves
What the program actually requires
Outcome engineer
Domain fluency + product thinking
Owns the question: what must this system actually do? Sits at the intersection of asset management knowledge and product design instinct. Probably a former operator who taught themselves to build, or a PM who spent years embedded in real estate.
  • Translates business outcomes into system behaviors
  • Defines what “correct” looks like for each decision type
  • Sets the bar for signal vs. noise
  • Validates prototypes with domain operators
Knowledge architect
Ontology + graph + meaning
Owns: how is reality encoded? Exists most frequently in biotech and defense — industries that solved this problem earlier. Rare in real estate. Builds the ontology, forces the governance conversations the organization has never had.
  • Constructs the entity graph and relationship model
  • Forces explicit answers to implicit governance questions
  • Ensures the ontology includes the human context layer
  • Partners with legacy IT to preserve institutional knowledge
Forward-deployed engineer
Full-stack + AI/agent fluency
The rarest archetype. Can ship production code and sit across from a CFO and earn their trust in the same week. Owns: build it where the work is. Works embedded in operations, not isolated in engineering.
  • Produces working prototypes in days, not months
  • Builds against real data from day one
  • Wires agents into actual workflows, not demos
  • Leaves artifacts that the organization can maintain
Trust & operations lead
Risk + audit + change management
Owns: does the org actually adopt? Easiest to find because it maps to existing change management and internal audit profiles — but must be genuinely interested in AI systems, not just governance theater.
  • Maps stakeholder resistance before it calcifies
  • Designs the trust evidence that makes pilots unfalsifiable
  • Translates agent behavior into language auditors accept
  • Builds the COE that survives the program team
What domain experts do in the new model
Domain experts stay essential — but as validators, not drivers. Their job shifts from defining requirements to stress-testing prototypes. They review working software, not Word documents. And critically: their time must be structurally committed — not “we’ll find time.”
04 · Program structure
Three gates, not one big bang
The reason most programs of this kind fail: funded as one large initiative with a delivery date 18 months out. By month six, requirements have shifted, the vendor selection looks wrong, and political will to change course is gone. Gates force honest evaluation before commitment deepens.
Change management · starts day one, runs concurrent throughout
Executives — protect budget + time
The sponsor who must become a champion

Small Frame the burning platform — not “cool technology” but “cost of waiting another year.”

Medium Gate discipline: working software at each review, not slides. Protect the team from scope expansion before Gate 1 clears.

Large Shift from sponsor to champion. When they mention the program unprompted, the threshold has been crossed.

Domain operators — the people whose work changes
Find the willing. Make them unfalsifiable.

Small Find the person who hates their current tool and has credibility. Not the loudest critic — the quiet builder.

Medium Pilot cohort embedded in live work — not a lab. Their feedback shapes the artifact re-imagination in real time.

Large Pilot becomes peer pressure. A room of skeptics watching a colleague do in 10 minutes what took 3 days. That moment is unfalsifiable.

Legacy IT + data — the team whose stack changes
Co-build or they will resist. Give them the keys.

Small Map who owns what. The person who built the load-bearing Excel becomes an architect — not a casualty.

Medium Co-build posture: their data gets a better home, not their systems get replaced.

Large IT owns the COE. Give them the keys before it feels completely safe. That’s how it sticks permanently.

Concurrent with workflow activation; neither ships without the other

Agentic systems only get adopted when operators can see why a recommendation was made. Decision lineage must be human-readable, not just machine-auditable. Cannot be bolted on after the fact.

Gate 2 · Expand / contract
Evidence: adoption rate + decision accuracy across pilot cohort
Technical infrastructure · capability audit first; all else concurrent once ontology is grounded
Capability audit — must happen first
Everything in the program gates on this. No exceptions.

Connect to existing systems. Extract signal about where operational friction lives. Produce a capability gap map grounded in actual data flows — not stated preferences. Becomes the anchor for every vendor conversation and foundation for the ontology.

Ontology + knowledge graph — entities, edges, meaning
Sequential after audit; integration hub runs concurrent

Taxonomies are trees. Agents need graphs. Building the ontology forces the organization to answer questions it has never had to make explicit. These governance gaps must be resolved here — the agent-ready data layer cannot be built until this is grounded.

Integration hub — connect what exists, don’t replace it
Concurrent with ontology build

Yardi, MRI, Excel, SharePoint, email — the sources of operational truth. The ontology determines what fields mean; the hub determines where they come from. These two workstreams run in parallel and converge at the agent-ready data layer.

Small
Capability audit + stakeholder mapping + prototype sprint. Days to weeks. People-cost dominated. No vendor selection yet.
Medium
Ontology v1 + integration hub + artifact re-imagination + pilot cohort. Weeks to a few months. Mix of people and infrastructure cost.
Large
Agent-ready data layer + scaled adoption + COE stood up. Several months onward. Mix shifts to OpEx. ROI measurable.
